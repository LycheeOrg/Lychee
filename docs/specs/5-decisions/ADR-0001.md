## Context

When setting up a local Docker Compose environment to develop and test Feature 029 (Camera Capture), the `docker-compose.yaml` inherited from the upstream LycheeOrg project included the following security hardening options applied via a `x-base-lychee-setup` YAML anchor shared across all services, and also directly on the `lychee_db` service:

```yaml
security_opt:
  - no-new-privileges:true
  - seccomp:unconfined
cap_drop:
  - ALL
cap_add:
  - CHOWN
  - SETUID
  - SETGID
  - NET_BIND_SERVICE
tmpfs:
  - /tmp
```

On Ubuntu hosts with AppArmor enabled, this combination caused two classes of failures:

1. **Service startup failure (`exec â€¦ operation not permitted`):** `no-new-privileges:true` combined with `cap_drop: ALL` / selective `cap_add` prevented the MariaDB and FrankenPHP container entrypoints from calling `setuid`/`setgid` to switch from root to the service user. Both `lychee_db` and `lychee_api` exited immediately with "operation not permitted".

2. **Unremovable containers (`cannot stop container: permission denied`):** AppArmor's default Docker profile blocked the Docker daemon from sending `SIGTERM`/`SIGKILL` to containers that had been started with restricted capabilities. The containers became impossible to stop or remove through normal Docker commands, requiring `sudo kill -9` of the container PIDs directly.

## Decision

For the local development Docker Compose configuration:

1. Remove `no-new-privileges:true` from both `x-base-lychee-setup` and `lychee_db`.
2. Remove `cap_drop: ALL` and `cap_add` lists from both.
3. Remove `tmpfs: [/tmp]` from both (not required for local dev).
4. Add `security_opt: apparmor=unconfined` to both to prevent AppArmor from blocking container signals.
5. Retain `seccomp:unconfined` (already present; keeps syscall filtering relaxed for dev).

## Consequences

**Positive:**
- Services start and run normally on Ubuntu with AppArmor.
- Containers can be stopped and removed through standard Docker commands.
- Developer iteration loop is unblocked.

**Negative:**
- The local development environment runs with a reduced security posture compared to production. Container processes run with the default Docker capability set rather than a hardened minimal set.
- AppArmor confinement is disabled for these containers.

**Mitigations:**
- This change applies **only** to the local development `docker-compose.yaml`. It must not be carried forward to any production deployment manifests (Kubernetes, Swarm, production Compose files).
- Production deployments should restore `cap_drop: ALL`, granular `cap_add`, and AppArmor/seccomp profiles appropriate to the target host OS.
- The upstream `docker-compose.yaml` security configuration should be treated as the production reference and reviewed before any deployment.

## Alternatives Considered

| Alternative | Reason rejected |
|-------------|-----------------|
| Fix capability set to allow entrypoint `setuid` (add `SETUID`/`SETGID`) | The upstream file already includes `SETUID`/`SETGID` in `cap_add`; the failure persisted, indicating `no-new-privileges:true` was the blocking constraint even with those caps present. |
| Disable AppArmor system-wide | Too broad; would affect all processes on the host. |
| Use a custom AppArmor profile for Docker | Appropriate for production hardening but disproportionate effort for a local dev environment. |
| Switch to Podman (rootless containers) | Not installed on the development machine; out of scope for this feature. |
