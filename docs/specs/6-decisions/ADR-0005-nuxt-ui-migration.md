# ADR-0005: Replace PrimeVue with Nuxt UI (standalone Vue mode)

- **Status:** Accepted, partially superseded by **ADR-0006**
- **Date:** 2026-07-02
- **Related features/specs:** Feature 049 (docs/specs/4-architecture/features/049-nuxt-ui-migration/spec.md)
- **Related open questions:** Q-049-01, Q-049-02, Q-049-03
- **Amendment note:** Decision item 1 below (in-place, file-by-file migration with single-bundle transitional coexistence) is superseded by ADR-0006's dual-tree, feature-flag-gated cutover strategy (Q-049-04). The Q-049-01 sizing decision (full scope, tracked to completion under one roadmap entry), Q-049-02 (icon parity via `@iconify-json/prime`), and Q-049-03 (drop ripple) resolutions below are unaffected and still govern the feature — only *how* the two libraries coexist during implementation changed, not *what* gets built or *how much*.

## Context

Lychee's frontend (`resources/js/`, Vue 3 + TypeScript + Vite + Tailwind CSS v4) has built its entire UI on PrimeVue (`primevue`, `@primeuix/themes`, `tailwindcss-primeui`, `primeicons`) since the app's inception. A codebase inventory (2026-07-02) found PrimeVue imported in 235 of 286 frontend files (~82%), with the largest costs concentrated in Tailwind utility classes generated by `tailwindcss-primeui` (197 files), a ~500-line custom theme preset with no equivalent in any replacement library, and PrimeVue-specific pass-through styling APIs (36-42 files).

The user requested migrating this UI layer to Nuxt UI (`@nuxt/ui`), a component library built on Reka UI primitives and Tailwind CSS v4 that, since its v3 release, supports standalone usage in a plain Vue 3 + Vite application without requiring the full Nuxt meta-framework. This aligns with the app's existing Tailwind v4 investment and avoids a much larger, higher-risk migration to the Nuxt framework itself (which would also require SSR/routing/build-tooling changes far beyond a UI-component swap).

Three architecturally significant sub-decisions needed resolution before planning could proceed (captured in [open-questions.md](../4-architecture/open-questions.md) as Q-049-01/02/03):
- **Sizing:** whether to plan the full 235-file removal as one feature, split it across several features, or commit only to a foundational subset.
- **Icons:** whether to preserve the existing PrimeIcons visual set (Iconify has a published `prime` collection mirroring PrimeIcons 1:1) or adopt Nuxt UI's default Lucide icon set as part of the migration.
- **Interaction parity:** PrimeVue's ripple click effect and `v-focustrap` directive have no Nuxt UI/Reka UI equivalent — Reka UI traps focus internally within its own primitives instead.

Affected modules: frontend only. No backend (PHP), REST, or CLI changes.

## Decision

Replace PrimeVue with Nuxt UI in standalone Vue mode across the entire main app bundle (everything reachable from `resources/js/app.ts`), as a single feature (Feature 049) tracked to full completion under one roadmap entry, with PrimeVue and Nuxt UI coexisting only transitionally during implementation.

Specifically:
1. **Sizing (Q-049-01):** One feature, many grouped increments (15 phases / 45 increments in plan.md), tracked to full completion. The feature is not considered done — and the roadmap entry does not move to Completed — until zero PrimeVue imports remain and `primevue`/`@primeuix/themes`/`tailwindcss-primeui`/`primeicons` are removed from `package.json` (FR-049-18, the hard completion gate).
2. **Icons (Q-049-02):** Preserve icon parity via the Iconify `prime` collection (`@iconify-json/prime`, published on npm, mirrors PrimeIcons 1:1). No icon redesign is bundled into this migration; a future visual refresh (e.g. adopting Lucide) is explicitly deferred to an optional follow-up feature.
3. **Interaction parity (Q-049-03):** Drop the ripple click effect entirely with no replacement; rely on Reka UI's built-in focus-trapping (replacing the removed `v-focustrap` directive) for modals, drawers, and other overlay primitives.
4. Two new architectural seams are introduced to fill gaps Nuxt UI does not cover natively: `useAppToast()` (centralizes the 119 previously-direct `useToast()` call sites behind one composable) and `useConfirmDialog()` + `ConfirmModalHost.vue` (replaces PrimeVue's imperative `ConfirmationService`/`useConfirm()`, which has no Nuxt UI built-in equivalent, with a Promise-based composable backed by a singleton `<UModal>`).
5. The embeddable widget bundle (`resources/js/embed/`, built via `vite.embed.config.ts`) is explicitly out of scope — it has zero PrimeVue coupling today and must remain dependency-free of both PrimeVue and Nuxt UI.

## Consequences

### Positive
- A single, coherent completion criterion (FR-049-18) prevents the codebase from settling into a permanently mixed-UI-library state, which would otherwise double the long-term maintenance burden (two theme systems, two icon sets, two component APIs).
- Decoupling the icon-set decision (parity now, redesign later) keeps this migration mechanical and low-risk, and keeps a future visual refresh independently reviewable.
- The existing Pinia-store-driven modal-visibility pattern (`ModalsState.ts`) and the absence of an imperative PrimeVue `DialogService` in the current codebase both turn out to already match Nuxt UI's declarative `<UModal v-model:open>` model closely, lowering migration risk for the highest-frequency interactive component after `Button`.
- Reka UI's built-in accessibility (ARIA roles, keyboard navigation, focus management) is expected to be equal to or better than the current hand-wired PrimeVue + directive setup.

### Negative
- This is among the largest single features in the project's history by file-touch count (235 files vs. the previous largest at 76 tasks/9 increments for a much narrower slice of the codebase), spanning an estimated 45 increments across 15 phases.
- PrimeVue and Nuxt UI necessarily coexist for the full duration of implementation (potentially spanning many sessions), during which two component libraries, two theme systems, and (partially) two icon systems are all present in the dependency tree and bundle.
- Nuxt UI's standalone-Vue-mode usage pattern is newer and less battle-tested than its primary (full-Nuxt) integration path; any gaps discovered during migration may require workarounds documented in plan.md's Follow-ups section.
- Several PrimeVue components have no direct Nuxt UI equivalent (`Toolbar`, `DataTable`'s slot-based column API, `VirtualScroller`, `Timeline`, `Stepper`, `ScrollTop`, `Inplace`, `SpeedDial`) and require bespoke composition or a structural rewrite rather than a mechanical prop-for-prop swap.
- Dropping the ripple effect is a small but visible interaction-feel change users may notice.

## Alternatives Considered

**Sizing (Q-049-01):**
- **A (chosen):** One feature, full scope, tracked to completion.
- **B:** Foundation feature plus follow-up features per component-family area — rejected because it leaves the app in an indefinitely mixed-library state between features with no committed timeline to close the gap, and multiplies roadmap/spec overhead for what is one logical migration.
- **C:** Foundation-only with the remainder deferred to backlog — rejected for the same reason as B, with higher risk since there is no committed follow-up at all.

**Icons (Q-049-02):**
- **A (chosen):** Icon parity via `@iconify-json/prime`.
- **B:** Redesign onto Nuxt UI's default Lucide set as part of this migration — rejected because it bundles a subjective, per-icon design decision (107 distinct icons) into what should be a mechanical library swap, inflating review risk and blocking clean ≤90-minute increments.

**Interaction parity (Q-049-03):**
- **A (chosen):** Drop ripple entirely; rely on Reka UI's built-in focus-trap.
- **B:** Build and maintain a custom ripple directive — rejected as new code to maintain indefinitely for a purely cosmetic effect, with re-verification risk against Nuxt UI's internal DOM structure across future version upgrades.

## Security / Privacy Impact

None. This is a frontend UI-component-library migration with no changes to authentication, authorization, data handling, secret management, or attack surface. Reka UI's primitives are the same class of client-side UI library as PrimeVue (no new third-party data flows, no new network calls).

## Operational Impact

- No backend/infrastructure operational impact — no new services, jobs, or deployment steps.
- Bundle size should be monitored during migration (NFR-049-04 in the feature spec) since Nuxt UI and PrimeVue coexist transitionally; the embeddable widget bundle (`public/embed/lychee-embed.js`) is specifically guarded against accidental growth since it ships to third-party host pages.
- No automated frontend regression test suite exists today (confirmed during spec drafting) and none is introduced by this migration; verification throughout is manual/browser-based per the project's standard practice for frontend changes, which increases the operational risk of an undetected visual/behavioral regression relative to a codebase with component-test coverage. A follow-up feature to introduce Vitest coverage is noted in plan.md's Follow-ups section.
